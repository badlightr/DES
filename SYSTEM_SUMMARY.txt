ğŸ“ KEY ARCHITECTURAL DECISIONS

1. JWT-based Authentication
   WHY: Stateless, scalable, modern
   NOT: Session-based (server state complexity)
   NOT: OAuth (overkill for internal HR system)

2. Role Hierarchy Model
   WHY: Implicit permissions reduce configuration
   SCALES: Easily add new roles between hierarchy
   EXAMPLE: ADMIN > HR > MANAGER > SUPERVISOR > EMPLOYEE

3. Approval Chain per Department
   WHY: Different departments have different workflows
   SCALES: Add/remove departments without code changes
   NOT: Hardcoded approval levels

4. Attendance Log Integration Point
   WHY: Overtime MUST be based on actual attendance data
   PREVENTS: User fraud (claiming hours never worked)
   READY: For integration with any time-tracking system

5. Soft Deletes + Row Versioning
   WHY: Audit trail completeness
   PREVENTS: Data loss, allows recovery
   AUDITABLE: Every change tracked with hash chain

6. Transactional Overlap Check
   WHY: Prevent race conditions
   USES: Database-level FOR UPDATE locks
   NOT: Application-level checks (insufficient)

7. Idempotency Keys
   WHY: Network failures are inevitable
   RETRY-SAFE: Same key returns cached response
   PREVENTS: Double billing, duplicate approvals
